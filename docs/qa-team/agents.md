# מבנה סוכני QA

זהו המבנה התפעולי של הסוכנים המומחים בפרויקט. כל סוכן אחראי על זווית בדיקה אחרת, וה-Orchestrator מחבר את הממצאים לתוכנית תיקון אחת.

```text
QA Orchestrator
├── Product QA Agent
│   ├── קורא דרישות
│   ├── בודק התאמה להתנהגות מוצרית
│   └── מחזיר פערים לפי Requirement ID
├── UX/A11y Agent
│   ├── בודק חוויית משתמש
│   ├── בודק RTL, מובייל ונגישות בסיסית
│   └── מחזיר ממצאי UX עם חומרה והמלצה
└── Engineering Agent
    ├── בודק קוד, טיפוסים, hooks ו-Supabase
    ├── מריץ או מנתח את שער האיכות האוטומטי
    └── מתקן באגים ומוסיף בדיקות רגרסיה
```

## זרימת עבודה בין הסוכנים

1. **QA Orchestrator** מגדיר את היקף הסבב ובוחר דרישות לבדיקה.
2. **Product QA Agent** משווה את האפליקציה ל-`requirements.md`.
3. **UX/A11y Agent** עובר על שימושיות, מובייל, עברית/RTL ונגישות בסיסית.
4. **Engineering Agent** מריץ `npm run qa`, מנתח כשלים ומתקן באגים שסוכמו.
5. **QA Orchestrator** מאחד את הכול לדוח סבב ב-`.qa/review-output/`.

## גבולות אחריות

| סוכן | כן עושה | לא עושה |
| --- | --- | --- |
| QA Orchestrator | מתעדף, מאחד ומחליט על סטטוס הסבב | לא מתקן לבד לפני שיש ממצא ברור |
| Product QA Agent | בודק התאמה לדרישות ולזרימות מוצר | לא משנה קוד |
| UX/A11y Agent | בודק חוויית משתמש ונגישות בסיסית | לא מחליף בדיקת נגישות מקצועית מלאה |
| Engineering Agent | מתקן באגים ומוסיף בדיקות | לא משנה דרישות מוצר בלי תיעוד |
