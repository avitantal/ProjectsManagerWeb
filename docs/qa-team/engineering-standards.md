# סטנדרטי פיתוח לפרויקט

המסמך הזה מגדיר מה נחשב שינוי איכותי בפרויקט React/TypeScript/Vite/Supabase הזה.

## TypeScript And React

- העדיפו טיפוסים קיימים מ-`src/lib/supabase.ts` על פני טיפוסים מקומיים כפולים.
- שמרו קומפוננטות ממוקדות: UI בקומפוננטות, גישה לנתונים ב-hooks או פונקציות ייעודיות.
- אל תוסיפו state חדש אם אפשר לגזור את הערך מ-state קיים באמצעות `useMemo`.
- פעולות async צריכות לטפל במצב טעינה, כשל או לפחות לא להשאיר UI במצב שגוי.
- שמות פונקציות צריכים לתאר פעולה עסקית, לא רק אירוע טכני.

## Supabase

- שמות טבלאות נגזרים מ-`scope` רק במקומות מבוקרים.
- כל שינוי כתיבה צריך לרענן את הנתונים הרלוונטיים דרך callback קיים.
- אין להציג סודות, keys או פרטי סביבה בדוחות.
- טיפול בשגיאות Supabase צריך להיות גלוי למשתמש כאשר הפעולה נכשלה.

## UI

- שמרו על תמיכה מלאה בעברית ו-RTL.
- השתמשו ב-lucide-react עבור אייקונים קיימים.
- לא מוסיפים טקסט הסבר בתוך האפליקציה במקום UX ברור.
- כפתורי פעולה הרסנית צריכים להיות מובחנים ודורשים כוונה ברורה.
- טקסט חייב להישאר בתוך גבולות הכפתור/כרטיס גם במסך צר.

## Tests

- שינוי בפונקציית מיון מחייב בדיקה יחידתית.
- שינוי בטופס יצירה/עריכה מחייב בדיקת rendering ולפחות בדיקת submit אחת.
- שינוי בסטטוס משימה או פרויקט מחייב בדיקה של `closed_at` כאשר רלוונטי.
- שינוי ב-hook של Supabase מחייב mock שמוודא שם טבלה, filters ו-ordering.
- באג שתוקן צריך לקבל בדיקת רגרסיה ממוקדת.

## Definition Of Done

- הקוד עובר `npm run qa`.
- כל ממצא חדש מתועד או נסגר.
- הדרישה הרלוונטית עודכנה אם ההתנהגות השתנתה.
- אין שינויי עיצוב לא מכוונים במסכים שלא נגעו בהם.
- אין שינויי formatting רחבים ללא צורך.
