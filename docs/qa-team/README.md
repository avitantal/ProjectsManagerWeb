# צוות איכות מצומצם לפרויקט

מטרת התיקייה הזו היא להפוך בדיקות איכות לתהליך קבוע, ברור וניתן לשחזור. הצוות בנוי מארבעה סוכנים/תפקידים קטנים: מוביל איכות, בודק מוצר, בודק UX ונגישות, ומהנדס React/TypeScript/Supabase.

## מתי מפעילים את התהליך

- לפני מיזוג שינוי משמעותי.
- אחרי תיקון באג שמשפיע על זרימת משתמש.
- לפני פריסה ל-GitHub Pages.
- כשנוספת דרישה חדשה או משתנה התנהגות קיימת.

## מבנה הצוות

| תפקיד | אחריות | תוצר |
| --- | --- | --- |
| Orchestrator | מפעיל את הסבב, מאחד ממצאים ומתעדף תיקונים | דוח סיכום ותוכנית פעולה |
| Product QA | משווה את האפליקציה מול הדרישות | סטטוס לכל דרישה ופערים |
| UX/A11y Reviewer | בודק חוויית שימוש, RTL, מובייל ונגישות בסיסית | ממצאי UX עם המלצות |
| Engineering Reviewer | בודק קוד, בדיקות, טיפוסים, Supabase ותקינות build | תיקונים או ממצאים טכניים |

מבנה הסוכנים וזרימת העבודה ביניהם מתוארים בפירוט ב-`agents.md`.

## תהליך העבודה

1. קוראים את `requirements.md` ומוודאים שהדרישות מעודכנות.
2. עוברים על `acceptance-checklist.md` ומסמנים לכל דרישה: תקין, חלקי, חסר או לא נבדק.
3. עוברים על `ux-review-checklist.md` במסכי desktop ו-mobile.
4. מריצים:

   ```bash
   npm run qa
   ```

5. מתעדים ממצאים לפי `bug-report-template.md`.
6. אם יש תיקון, מוסיפים או מעדכנים בדיקה רלוונטית.
7. סוגרים סבב עם `review-report-template.md`.

## כלל החלטה

שינוי מוכן רק אם:

- אין ממצאי Critical או High פתוחים.
- `npm run qa` עובר או שיש סיבה מתועדת וברורה לכשל.
- כל דרישה שנגעה בשינוי נבדקה.
- יש בדיקה אוטומטית להתנהגות מרכזית או נימוק למה היא לא נוספה.
